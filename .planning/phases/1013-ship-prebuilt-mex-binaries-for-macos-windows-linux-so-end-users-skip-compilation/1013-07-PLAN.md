<objective>
Close the single gap from 1013-VERIFICATION.md: on a fresh macOS ARM64 Octave session (only repo root on path), `install()` must skip the "--- Compiling MEX files ---" banner when the stamp matches. Root cause: `mex_stamp.m` lives in `libs/FastSense/private/` and is invisible to `install.m` at repo root, so `stamp_matches_`'s try/catch silently returns false and `needs_build` falls through to rebuild. Compilation is avoided only by `build_mex.m`'s secondary mtime guard.

Fix (minimum viable, option b from VERIFICATION.md): **relocate `mex_stamp.m` from `libs/FastSense/private/` to `libs/FastSense/` (public scope)**. `install.m` already addpaths `libs/FastSense` at line 51, so the function becomes reachable with zero code change beyond the move. Also update the regression tests (MATLAB class + Octave function twin) to probe the correct platform binary — flat `.mexmaca64`/`.mexa64`/`.mexw64` on MATLAB, `private/octave-<tag>/binary_search_mex.mex` on Octave — so Test 4 asserts instead of auto-SKIPping under the Plan 03 subdir layout.

TDD: update the regression tests FIRST (they must fail on current code because stamp fast-path is broken), then move `mex_stamp.m` to make them pass.

Purpose: The primary gating mechanism (stamp-check fast path) becomes operative end-to-end. Users on a fresh clone see no compile banner when binaries match sources.

Output: `install('__probe_needs_build__')` returns `false` on fresh path state; regression suite asserts (does not skip) Test 4 on both runtimes.
