<objective>
Fix bug 260508-l2k: the dashboard time-slider preview and event-marker overlay are empty on tabs whose only data/event-bearing widgets are nested inside a GroupWidget.

Root cause: `DashboardEngine.computePreviewEnvelopeReturning_` and `computeEventMarkers` iterate `obj.activePageWidgets()` which returns top-level widgets only. They call `getPreviewSeries` / `getEventTimes` on the Group itself; the base implementations return `[]`, so nested FastSenseWidgets contribute nothing.

Fix: introduce a virtual `getNestedWidgets()` method on `DashboardWidget` (default `{}`), override on `GroupWidget` to return `Children` plus the union of all `Tabs{*}.widgets`, and add a depth-first flatten helper `flattenWidgetsForPreview_` on `DashboardEngine`. Use the helper at both iteration sites. Preserve per-tab semantics: flatten the *active page's* widget list, never `allPageWidgets()`.

Purpose: Restore preview-line and event-marker visibility on Pages 2/3/5/6 of the industrial demo (where data-bearing widgets live inside GroupWidgets) without regressing Pages 1/4 (top-level widgets).

Output: Patched DashboardWidget / GroupWidget / DashboardEngine, plus two new regression test cases.
</objective>

<bug_diagnosis>
GroupWidget's `Children` (cell of DashboardWidget) contain the actual data-bearing FastSenseWidgets. The engine never unwraps these for preview/marker collection. Existing precedent: GroupWidget.getTimeRange already recurses; we mirror that pattern at the engine level via a virtual method + flatten helper.

Live evidence:
- Page 2 "Feed Line" `activePageWidgets()` returns 5 top-level: GroupWidget, StatusWidget, BarChartWidget, DividerWidget, TextWidget — none implement getPreviewSeries.
- The GroupWidget contains 2 FastSenseWidgets in `Children` that DO have data and DO render plots.
- Pages 1, 4 (top-level FastSenseWidgets) work today — must not regress.
</bug_diagnosis>

<solution_recap>
1. Add virtual `getNestedWidgets()` on `DashboardWidget` (returns `{}`).
2. Override on `GroupWidget` to return `[Children, flatten(Tabs{*}.widgets)]` as a cell.
3. Add private `flattenWidgetsForPreview_(obj, widgets)` on `DashboardEngine`: depth-first walk, leaves are widgets whose `getNestedWidgets()` is empty.
4. Replace the two `ws = obj.activePageWidgets();` lines with `ws = obj.flattenWidgetsForPreview_(obj.activePageWidgets());`.
5. Add two regression test cases (one for preview, one for events).

The virtual-method approach is open-extension: future container widgets just override `getNestedWidgets()` without touching the engine. (Note: `MultiStatusWidget` does NOT have a public `Children` cell of DashboardWidget — its UI children are ephemeral handles, NOT subclassed widgets. Only `GroupWidget` overrides.)
</solution_recap>
